# Лаб 3 — ПДСЧ (псевдо-датчик случайных чисел)

**Crate:** `crates/lab_03_prng`
**Статус:** ✅ генератор готов — **22 теста** (16 unit/property/sanity + 6 интеграционных CLI), clippy clean. Прогон через NIST STS — оставлен пользователю (нужен ручной запуск собранной утилиты).

## Условие

> Реализовать приложение, генерирующее набор значений ПДСЧ (не менее 200 значений). Алгоритм реализации произвольный, необходимо рассказать плюсы и минусы выбранного алгоритма.
> Полученный набор значений сохранить в файл.
> Скачать из облака архив `sts-2_1_2.zip` из исходников скомпилировать утилиту проверки ПДСЧ и выполнить проверку получившейся последовательности.
> В облаке также размещён файл `nistspecialpublication800-22r1a.pdf` с документацией по стандартам NIST и описанием работы библиотеки проверки (см. главу 5).

## Выбранный алгоритм: xorshift64*

```
x ^= x << 13;
x ^= x >> 7;
x ^= x << 17;
return x * 0x2545F4914F6CDD1D;
```

**Плюсы**
- Очень быстрый, без умножений (кроме финального scrambling).
- Период $2^{64} - 1$ — более чем достаточно для 200 значений и для прогонов NIST STS.
- Простой код, легко защищается на лаб 5 (нет неинициализированной памяти, нет UB).
- Проходит большинство тестов TestU01 SmallCrush и базовые тесты NIST STS на разумных длинах.

**Минусы**
- **Не криптографически стойкий** — по 2-3 последовательным выходам легко восстановить состояние. Для шифрования не подходит.
- Тестовая структура NIST показывает локальные смещения в некоторых тестах (Approximate Entropy, Random Excursions) на коротких последовательностях.
- Один статический seed → детерминированная последовательность (но это нам как раз и нужно для воспроизводимости в лаб 4).

Альтернативы, которые я рассмотрел и отверг:
- LCG (linear congruential) — слишком короткий период младших бит, плохо проходит NIST.
- Blum-Blum-Shub — крипто-стойкий, но медленный, требует длинной арифметики и качественной генерации простых.

## Что попадает в файл

Бинарный поток битов: 8 байт на каждое 64-битное значение, big-endian. 200 значений = 1600 байт = 12 800 бит, ровно столько и требует STS для тестов вида Frequency и Block Frequency на стандартных параметрах.

Дополнительно сохраняется ASCII-вариант (`*.bits`) — последовательность `'0'`/`'1'`, удобная для лаб 4 и для просмотра.

## Реализация (план)

- `domain::prng::Xorshift64Star` — `new(seed)`, `next_u64()`.
- `domain::prng::stream::BitStream` — итератор над битами.
- `application::generate` — взять seed, сгенерировать $N$ значений, записать в файл.
- `infrastructure::file_sink` — пишет binary + ascii варианты.
- `presentation::cli` — `gen --count 200 --seed 0xDEADBEEFCAFEBABE --out artifacts/lab_03/seq`.

## Как запустить

```bash
just lab-03 gen --count 200 --seed 0xDEADBEEFCAFEBABE \
    --out-bin artifacts/lab_03/sequence.bin \
    --out-ascii artifacts/lab_03/sequence.bits
```

## Как прогнать через NIST STS 2.1.2

> Архив (sts-2_1_2.zip) предоставляется преподавателем «из облака». Альтернатива: [исходники с csrc.nist.gov](https://csrc.nist.gov/Projects/Random-Bit-Generation/Documentation-and-Software). Внутри — `experiments/` + `Makefile`.

```bash
# (один раз) распаковка и сборка
unzip sts-2_1_2.zip -d external/
cd external/sts-2.1.2/
make

# прогон на нашей последовательности
./assess 12800
# в интерактиве выбрать "0 - Input File", указать путь к ASCII-файлу нашей последовательности
```

Отчёт STS появится в `external/sts-2.1.2/experiments/AlgorithmTesting/finalAnalysisReport.txt`. Туда вкладываем выводы в `docs/explanations/lab_03_prng/sts-report.md`.

## Как протестировать

```bash
cargo test -p lab_03_prng
```

- Воспроизводимость: одинаковый seed → одинаковая последовательность.
- Длина выхода: при `count=200` файл имеет ровно 1600 байт.
- Базовая статистика: на 200 значениях ожидаемое количество единиц ≈ половина от 64×200 (Chi² в широких границах).
