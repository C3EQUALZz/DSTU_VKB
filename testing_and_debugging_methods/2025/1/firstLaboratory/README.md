## Используемая терминология

Согласно классической модели Goodenough/Hoare и определениям ISTQB:

| Термин | Значение |
|--------|----------|
| **Fault** (дефект) | Статический недостаток в коде — неверная конструкция, выражение, условие. |
| **Error** (ошибка) | Несоответствующее правильному внутреннее состояние программы, возникающее при выполнении дефектного участка. |
| **Failure** (отказ) | Внешне наблюдаемое неправильное поведение программы — несоответствие фактического результата ожидаемому. |

Тест-кейс «выполняет fault» означает, что входные данные **приводят к исполнению** дефектной строки. При этом он не обязательно превращает fault в failure — для этого должна ещё распространиться «error» до выхода функции и проявиться в наблюдаемом результате.

## Задание №1 (а): где находится fault и как его исправить

### Локализация дефекта

Дефектной является строка с условием:

```java
if (x[i] % 2 == 1 || x[i] > 0) {
```

Точка fault — **левая часть дизъюнкции** `x[i] % 2 == 1`. Цель этой проверки —
определить, является ли число нечётным. Но в Java оператор остатка `%` для
**отрицательных** операндов возвращает **отрицательный** остаток. Это
зафиксировано в JLS §15.17.3:

> The result of the binary `%` operator is such that `(a/b)*b + (a%b) == a`.

Из этого следует:

| Значение `x[i]` | `x[i] % 2` | `x[i] % 2 == 1`? |
|-----------------|-----------|------------------|
| `3` | `1` | `true` |
| `-3` | `-1` | **`false`** ← ошибка |
| `-1` | `-1` | **`false`** ← ошибка |
| `-7` | `-1` | **`false`** ← ошибка |

Следовательно, для любого **отрицательного нечётного** числа первое подусловие
ложно. Второе подусловие `x[i] > 0` для отрицательного числа также ложно.
Итог: метод **пропускает все отрицательные нечётные элементы**, хотя по
контракту в JavaDoc должен их учитывать («count of **odd** or positive
elements in `x`»).

### Способы исправления

Простейший и самый идиоматичный фикс — проверять остаток на «не равен нулю»:

```java
if (x[i] % 2 != 0 || x[i] > 0) {
```

`x % 2 != 0` истинно и для `+1`, и для `−1`, то есть для любого нечётного
целого, независимо от знака.

Альтернативные варианты, дающие тот же эффект:

```java
// 1. Использовать модуль остатка
if (Math.abs(x[i] % 2) == 1 || x[i] > 0) { ... }

// 2. Использовать Math.floorMod — он всегда возвращает неотрицательный остаток
if (Math.floorMod(x[i], 2) == 1 || x[i] > 0) { ... }

// 3. Битовая проверка младшего бита (только для чисел в дополнительном коде)
if ((x[i] & 1) == 1 || x[i] > 0) { ... }
```

В исходниках лабораторной (`OddOrPos.java`) исправленная версия вынесена в
отдельный метод `oddOrPosFixed`, чтобы можно было сравнивать поведение
обеих реализаций в тестах и в `Main`.

## Задание №1 (б): test cases, **не выполняющие** fault

Тест-кейс **не выполняет fault**, если на нём результат багованной реализации
**совпадает** с эталонной (исправленной), т.е. внешне ошибка не заметна.
Это возможно в нескольких ситуациях.

### Кейс 1. Пустой массив

| Параметр | Значение |
|----------|----------|
| Вход | `x = {}` |
| Ожидаемый результат | `0` |
| Фактический результат | `0` |
| Почему fault не сработал | Цикл `for` не выполняется ни одной итерации — дефектное условие никогда не вычисляется. |

### Кейс 2. Массив без отрицательных нечётных чисел

| Параметр | Значение |
|----------|----------|
| Вход | `x = {0, 1, 2, 3, 4, 5}` |
| Ожидаемый результат | `5` (учтены `1, 2, 3, 4, 5`) |
| Фактический результат | `5` |
| Почему fault не сработал | Для всех элементов либо `x[i] % 2 == 1` истинно (нечётные положительные `1, 3, 5`), либо срабатывает второе подусловие `x[i] > 0` (`2, 4`). Для нуля оба подусловия ложны, но это и есть правильное поведение. |

### Кейс 3. Все элементы — положительные нечётные

| Параметр | Значение |
|----------|----------|
| Вход | `x = {1, 3, 5, 7}` |
| Ожидаемый результат | `4` |
| Фактический результат | `4` |
| Почему fault не сработал | Дефектная ветка `x[i] % 2 == 1` для положительных нечётных работает **верно** — `%` возвращает `1`. Дополнительно «прикрывает» правая ветка `x[i] > 0`. |

### Кейс 4. Все элементы — отрицательные чётные

| Параметр | Значение |
|----------|----------|
| Вход | `x = {-2, -4, -6, -8}` |
| Ожидаемый результат | `0` |
| Фактический результат | `0` |
| Почему fault не сработал | Дефектное условие **выполняется** на каждой итерации, но всегда корректно даёт `false`: число не нечётное (`x % 2 == 0`) и не положительное. Дефект потенциально *достижим*, но не приводит к ошибочному состоянию (`reachability` есть, `infection` нет). |

### Кейс 5. Все нули

| Параметр | Значение |
|----------|----------|
| Вход | `x = {0, 0, 0}` |
| Ожидаемый результат | `0` |
| Фактический результат | `0` |
| Почему fault не сработал | `0 % 2 == 0`, `0 > 0` — false. Ситуация аналогична кейсу 4. |

> **Обобщение.** Fault не превращается в failure, если в массиве отсутствуют
> отрицательные нечётные элементы. Иначе говоря, чтобы тест **выполнил**
> fault и привёл к видимому отказу, входной массив **должен содержать хотя бы
> одно число вида `2k − 1`, где `k ≤ 0`** (например `−1`, `−3`, `−5`, …).

## Задание №2: Normal Boundary Value Testing для метода `area`

### Краткая теория

**Boundary Value Analysis (BVA)** — методика чёрного ящика, выбирающая
тестовые значения «по краям» области допустимых входов, где исторически
сосредоточена бо́льшая часть ошибок (off-by-one, неверные сравнения
`<` vs `<=`, неверная инициализация и т.п.).

Различают несколько форм BVA:

| Форма | Какие значения берутся для каждой переменной | Кол-во тест-кейсов (n переменных) |
|-------|----------------------------------------------|----------------------------------|
| **Normal BVT** | `min`, `min⁺`, `nom`, `max⁻`, `max` (только in-range) | **4·n + 1** |
| Robust BVT | те же + `min⁻`, `max⁺` (out-of-range) | 6·n + 1 |
| Worst-case BVT | декартово произведение 5 значений каждой переменной | 5ⁿ |
| Robust Worst-case BVT | то же, но 7 значений | 7ⁿ |

В Normal BVT действует принцип «**single fault assumption**»: предполагается,
что отказ вызывает изменение **одной** переменной за раз. Поэтому в каждом
кейсе одна переменная варьируется через все 5 граничных значений, а
остальные **фиксируются на nominal**.

### Параметры тестирования метода `area`

Реализация метода: [`Quadrilateral.java`](src/main/java/org/cequalz/Quadrilateral.java).
Для прямоугольника с двумя независимыми сторонами площадь = `side1 · side2`.

> **Замечание к терминологии.** В строгой геометрии «regular quadrilateral»
> — это квадрат. Однако спецификация задаёт *две независимых* стороны, что
> противоречит определению квадрата. Это типичная неточность учебных
> заданий: фактически тестируется формула площади прямоугольника
> `S = side1 · side2`.

Граничные точки (одинаковы для обеих переменных, так как диапазоны совпадают):

| Метка | Значение |
|-------|---------:|
| `min`  |   `0` |
| `min⁺` |   `1` |
| `nom`  |  `50` |
| `max⁻` |  `99` |
| `max`  | `100` |

Число тест-кейсов: **4·n + 1 = 4·2 + 1 = 9**.

### Таблица тест-кейсов (Normal BVT)

| TC # | side1 | метка side1 | side2 | метка side2 | Ожидаемый `area` |
|------|------:|-------------|------:|-------------|-----------------:|
| TC-1 |   `0` | `min`       |  `50` | `nom`       |            **0** |
| TC-2 |   `1` | `min⁺`      |  `50` | `nom`       |           **50** |
| TC-3 |  `50` | `nom`       |  `50` | `nom`       |         **2500** |
| TC-4 |  `99` | `max⁻`      |  `50` | `nom`       |         **4950** |
| TC-5 | `100` | `max`       |  `50` | `nom`       |         **5000** |
| TC-6 |  `50` | `nom`       |   `0` | `min`       |            **0** |
| TC-7 |  `50` | `nom`       |   `1` | `min⁺`      |           **50** |
| TC-8 |  `50` | `nom`       |  `99` | `max⁻`      |         **4950** |
| TC-9 |  `50` | `nom`       | `100` | `max`       |         **5000** |

Графически это «крест» из двух пробегов по 5 точек, пересекающихся в
центральной точке (TC-3):

```
side2
 100 ┤                       • TC-9
  99 ┤                       • TC-8
  50 ┤ • • • • •  ← TC-1..TC-5 (центр = TC-3)
   1 ┤                       • TC-7
   0 ┤                       • TC-6
     └─────────────────────
        0  1   50    99 100  side1
```

### Дополнительные кейсы вне Normal BVT (для полноты отчёта)

Хотя строго `Normal BVT` ограничен **только in-range** значениями,

| RC-2 | `101` | `50`  | `IllegalArgumentException` |
| RC-3 | `50`  | `-1`  | `IllegalArgumentException` |
| RC-4 | `50`  | `101` | `IllegalArgumentException` |

## Выводы

### По Заданию №1

1. **Fault** располагается в условии цикла, в подвыражении `x[i] % 2 == 1`.
   Причина — некорректное предположение о знаке остатка от деления на 2
   в Java для отрицательных чисел.
2. **Минимальный фикс** — заменить `== 1` на `!= 0`. Это не меняет
   корректного поведения для положительных и нулевых значений и
   восстанавливает корректность для отрицательных.
3. **Тестируемость:** проверить только что-то вроде `{1, 2, 3}` или
   `{0}` — недостаточно. Чтобы поймать такой класс ошибок, в тест-наборе
   обязательно должны быть **отрицательные нечётные** значения, в том
   числе граничные (`-1`, `Integer.MIN_VALUE + 1`).

### По Заданию №2

1. Метод имеет две независимые целочисленные входные переменные с
   одинаковым диапазоном `[0, 100]`, поэтому **Normal BVT** даёт
   **4·2 + 1 = 9** тест-кейсов.
2. Применён принцип «single fault assumption»: одна переменная
   пробегает значения `{min, min⁺, nom, max⁻, max}`, другая
   зафиксирована на `nom = 50`.
3. Normal BVT покрывает только «легальные» входы; для проверки
   корректной обработки нарушенных пред-условий понадобятся
   дополнительные кейсы из Robust BVT (`min⁻ = -1`, `max⁺ = 101`).
