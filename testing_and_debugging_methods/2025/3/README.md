## Краткая теория: эквивалентные классы

Разбиение области входных значений на **эквивалентные классы** (Equivalence
Class Partitioning, ECP) основано на отношении эквивалентности, которое
по определению удовлетворяет трём свойствам:

| Свойство | Формально | Смысл |
|----------|-----------|-------|
| Рефлексивность   | `a ~ a`                                         | элемент эквивалентен сам себе |
| Симметричность   | `a ~ b ⇒ b ~ a`                                 | направление отношения не важно |
| Транзитивность   | `a ~ b ∧ b ~ c ⇒ a ~ c`                         | классы не пересекаются |

В тестировании это даёт **разбиение** области входов на непересекающиеся
классы, каждый из которых, как предполагается, обрабатывается программой
одинаково. Из каждого класса достаточно взять одного «представителя» —
это и есть основа Equivalence Class Testing (ECT).

Различают четыре основные формы ECT:

| Метод | Кол-во тест-кейсов | Что гарантирует |
|-------|--------------------|-----------------|
| **Weak Normal ECT** (WN) | `max(\|Cᵢ\|)` | каждый валидный класс каждой переменной покрыт ≥ 1 раз |
| Strong Normal ECT (SN)   | `∏ \|Cᵢ\|`   | декартово произведение валидных классов |
| Weak Robust ECT (WR)     | WN + 1 невалидный класс на переменную | расширение WN out-of-range |
| Strong Robust ECT (SR)   | декартово произведение всех классов | максимальное покрытие |

**Improved Weak Normal Equivalence Class Testing (IWN)** — улучшение WN:
помимо требования «каждый класс покрыт хотя бы раз» добавляется
ограничение, что **в одном и том же тест-кейсе не должны одновременно
встречаться представители только одних и тех же классов** (стремится к
большему разнообразию комбинаций). Число тестов остаётся `max(|Cᵢ|)`.

## Задание №1 — выбор корректного набора тест-кейсов IWN

### Условие

Функция верифицирует пользователя по паре `(username, password)`. Заданы
разбиения:

| Переменная | Класс | Диапазон длин |
|------------|------|---------------|
| `username` | uA   | 8–10 символов |
| `username` | uB   | 11–12 символов |
| `username` | uC   | 13–16 символов |
| `password` | pA   | 8–11 символов |
| `password` | pB   | 12–16 символов |

## Задание №2 — Unit-тестирование `next_day` + поиск бага

### Условие

Задана функция `next_day(...)` в пакете `services`. По рисунку 5
вызов выглядит так:

```python
from services import nextday
year  = int(input("Введите год: "))
month = int(input("Введите месяц: "))
day   = int(input("Введите день: "))
print(f"Следующим днем будет: {nextday.next_day(year, month, day)}")
```

При вводе `year=2023, month=11, day=10` функция возвращает **`None`**.

### Локализация бага

Сигнатура исходной функции (Рисунок 3):

```python
def next_day(day, month, year):   # порядок параметров: day → month → year
    ...
```

Вызов (Рисунок 5):

```python
nextday.next_day(year, month, day)   # порядок: year → month → day
```

Параметры **передаются в противоположном порядке**. Что происходит при
вводе `2023, 11, 10`:

| Имя при вызове | Передано | Имя в функции |
|----------------|---------:|---------------|
| year  | `2023` | day  ← попадает значение года |
| month | `11`   | month |
| day   | `10`   | year ← попадает значение дня |

Дальше функция вычисляет `max_days = days_in_month(11, 10) = 30`,
проверяет `day (=2023) > max_days (=30)` — условие истинно, поэтому
возвращается `None`. Это и наблюдаемая ошибка.

### Способы исправления

Существует **два** валидных варианта:

| Вариант | Что меняем | Минус |
|---------|------------|-------|
| **A** — поменять порядок параметров функции на `(year, month, day)` | один файл, исправлена *семантика* | если кто-то ещё вызывает `next_day(day, month, year)` — он сломается |
| **B** — поменять порядок аргументов в коде запуска | один файл, исправлен вызов | оригинальный «странный» порядок остаётся в API |

В данной лабораторной выбран **вариант A** — это естественный порядок
(от старшего поля к младшему, как в `datetime.date(year, month, day)`),
и он совпадает с тем, как функция фактически вызывалась в Рисунке 5.

> Исправленная функция реализована в
> [`src/third_laboratory/services/nextday.py`](src/third_laboratory/services/nextday.py).
> Оригинальная (дефектная) версия сохранена в
> [`nextday_legacy.py`](src/third_laboratory/services/nextday_legacy.py)
> для тестов-«пугалок».

### Реализация `next_day(year, month, day)`

```python
def next_day(year: int, month: int, day: int) -> Optional[tuple]:
    # Тип: только int, bool отдельно отвергаем
    for v in (year, month, day):
        if isinstance(v, bool) or not isinstance(v, int):
            return None

    if year < 1 or month < 1 or month > 12 or day < 1:
        return None

    max_days = days_in_month(month, year)
    if max_days is None or day > max_days:
        return None

    if day < max_days:
        return (year, month, day + 1)
    if month < 12:
        return (year, month + 1, 1)
    return (year + 1, 1, 1)
```

### Эквивалентные классы для `next_day`

#### По переменной `year`

| Класс | Условие | Представители |
|-------|---------|---------------|
| **Y1** | невисокосный год (`year % 4 != 0` или (`year % 100 == 0` и `year % 400 != 0`)) | 2023, 2025, 1900, 2100 |
| **Y2** | високосный, кратный 4 не 100 | 2024, 2020, 1996 |
| **Y3** | високосный, кратный 400 | 2000, 2400, 1600 |
| Y4 (invalid) | `year < 1` | 0, -1, -100 |
| Y5 (invalid) | не `int` | `None`, `"2023"`, `2023.0`, `True` |

#### По переменной `month`

| Класс | Условие | Представители |
|-------|---------|---------------|
| **M1** | месяц с 31 днём | 1, 3, 5, 7, 8, 10, 12 |
| **M2** | месяц с 30 днями | 4, 6, 9, 11 |
| **M3** | февраль | 2 |
| M4 (invalid) | `month < 1` | 0, -1 |
| M5 (invalid) | `month > 12` | 13, 100 |
| M6 (invalid) | не `int` | `None`, `"5"`, `5.0` |

#### По переменной `day`

| Класс | Условие | Что делает функция |
|-------|---------|--------------------|
| **D1** | `1 ≤ day < days_in_month(month, year)` | `day + 1` в том же месяце |
| **D2** | `day == days_in_month` и `month ≠ 12` | переход на 1-е число следующего месяца |
| **D3** | `day == 31` и `month == 12` | переход на 1 января следующего года |
| D4 (invalid) | `day < 1` | `None` |
| D5 (invalid) | `day > days_in_month` | `None` |
| D6 (invalid) | не `int` | `None` |

## Установка и запуск

### Требования

| Инструмент | Версия |
|------------|--------|
| Python | **3.14+** (указано в `.python-version`) |
| [`uv`](https://docs.astral.sh/uv/) | актуальная |

### 1. Установка `uv`

macOS / Linux:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
# или
brew install uv
```

Windows (PowerShell):

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

### 2. Установка зависимостей

```bash
cd testing_and_debugging_methods/2025/3
uv sync
```

`uv` сам поставит Python 3.14 (если нужно), создаст `.venv/`, поставит
`pytest` и установит пакет `third_laboratory` в режиме editable.

### 3. Запуск всех тестов

```bash
uv run pytest
```

Ожидаемый результат:

```
115 passed in 0.05s
```

### 4. Полезные варианты запуска

С подробным выводом:

```bash
uv run pytest -v
```

Запуск только тестов основной (исправленной) функции:

```bash
uv run pytest tests/unit/test_nextday.py
```

Запуск только тестов, демонстрирующих баг из задания:

```bash
uv run pytest tests/unit/test_legacy_bug.py -v
```

Запуск конкретного класса тестов (например, на високосные годы):

```bash
uv run pytest tests/unit/test_nextday.py::TestEquivalenceValidYear -v
```

Фильтрация по имени:

```bash
uv run pytest -k "leap" -v
```

### 5. Прямой запуск функции

```bash
uv run python -c "from third_laboratory.services import next_day; print(next_day(2023, 11, 10))"

uv run python -c "from third_laboratory.services.nextday_legacy import next_day; print(next_day(2023, 11, 10))"
```

## Выводы

### По Заданию №1

* Улучшенная слабая нормальная эквивалентность (**IWN ECT**) требует
  `max(|Cᵢ|)` тестов с обязательным присутствием каждого класса каждой
  переменной хотя бы в одном кейсе.
* Из 4 предложенных наборов этому критерию удовлетворяет **только
  вариант 4**: он содержит представителей всех трёх классов `username`
  (uA, uB, uC) и обоих классов `password` (pA, pB).
* Варианты 1, 2 и 3 нарушают покрытие хотя бы одного класса.

### По Заданию №2

* **Локализованный баг** — рассогласование порядка параметров между
  определением функции (`day, month, year`) и её вызовом
  (`year, month, day`). Это типовая ошибка, которую статический
  type-checker не отлавливает, потому что все три параметра одного
  типа (`int`).
* Минимально-инвазивная правка — приведение сигнатуры к
  `(year, month, day)`. Это совпадает с естественным порядком (от
  старшего к младшему) и тем, как функция реально вызывалась.
* Дополнительно ужесточена проверка типа (`bool` отвергается явно),
  чтобы исключить путаницу с подклассом `int`.
* Для покрытия использованы эквивалентные классы по трём переменным:
  `year` (Y1–Y3 валидных, Y4–Y5 невалидных), `month` (M1–M3 / M4–M6),
  `day` (D1–D3 / D4–D6). Получено **115** pytest-тестов, все зелёные.
