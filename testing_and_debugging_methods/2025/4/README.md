# Лабораторная работа №4

**Дисциплина:** Методы тестирования и отладки программного обеспечения
**Тема:** Тестирование полносвязной архитектуры искусственной нейронной сети.

---

## Цель работы

Создать систему компьютерного зрения, определяющую тип геометрической
фигуры (треугольник, круг, квадрат), и провести серию экспериментов по
тестированию точности модели и подбору гиперпараметров с использованием
полносвязной (Fully-Connected / MLP) архитектуры.

> **Стек:** PyTorch вместо TensorFlow/Keras (по согласованию).
> Архитектура от этого не меняется — фреймворк просто заменён на
> эквивалентный.

---

## Что варьируется (по заданию)

| Гиперпараметр | Значения |
|---------------|---------|
| Число нейронов в скрытом слое | **10**, **100**, **5000** |
| Активация в скрытом слое      | **relu**, **linear** |
| `batch_size`                  | **10**, **100**, **1000** |

Всего: **3 × 2 × 3 = 18 экспериментов**.

Метрики, выводимые на экран по каждому эксперименту:
* `accuracy` — доля верно классифицированных тестовых изображений;
* `MSE` — среднеквадратичная ошибка `softmax(logits)` против one-hot
  истинных меток;
* `MAE` — среднеабсолютная ошибка по той же постановке.

## Датасет

В исходном задании предлагалось загрузить датасет из Google Colab.
Чтобы лаба была **воспроизводимой** и не зависела от внешнего ресурса,
датасет генерируется программно через Pillow (см.
[`src/fourth_laboratory/dataset.py`](src/fourth_laboratory/dataset.py)):

* три класса: `circle` (0), `square` (1), `triangle` (2);
* изображения grayscale `28 × 28` пикселей (как MNIST);
* для каждой фигуры случайно варьируется размер, положение центра,
  угол поворота (квадрат/треугольник), толщина контура (или заливка),
  плюс лёгкий гауссовский шум фона;
* размер по умолчанию: **800** изображений каждого класса в train,
  **200** — в test (итого 2400 / 600);
* всё детерминировано по `seed` — повторный запуск даёт идентичный
  датасет.

Это даёт чистый, сбалансированный, проверяемый на лету корпус —
достаточный для демонстрации различий между конфигурациями.

## Архитектура

Модель ([`src/fourth_laboratory/model.py`](src/fourth_laboratory/model.py)):

```
Input  : (B, 28, 28)
Flatten        ─► (B, 784)
Linear(784, H) ─► (B, H)
Activation     ─► (B, H)        # ReLU или Identity (linear)
Linear(H, 3)   ─► (B, 3)        # логиты по 3 классам
```

Число параметров: `784·H + H + H·3 + 3` (для H=5000 это ~3.94 млн).

Функция потерь — `CrossEntropyLoss` (включает softmax + NLL).
Оптимизатор — `Adam`, learning rate `1e-3`. Эпох — `10` (по умолчанию).

### Краткие наблюдения

1. **ReLU стабильно лучше Linear.** На любой ширине слоя и при любом
   `batch_size` ReLU даёт точность выше: разница достигает 19 п.п.
   (`5000/linear/10` vs `5000/relu/100`). Это закономерно: с `linear`-
   активацией двухслойная сеть математически эквивалентна одному
   линейному преобразованию, теряя выразительность.
2. **Большой слой помогает, но не безусловно.** Прирост от 100 → 5000
   нейронов виден только в связке с `relu` и `batch_size=100`.
   Маленький `batch_size=10` с 5000 нейронами проигрывает 100-нейронной
   сети с тем же batch (`0.8267` против `0.8500`) — слишком много
   шума градиента на огромных весах.
3. **Большой `batch_size=1000` стабильно проигрывает.** При наших
   2400 примерах батч в 1000 — это всего 2–3 шага оптимизатора за
   эпоху, чего недостаточно. Лучше всего работает `batch_size=100`
   (3–4 десятка шагов на эпоху, при этом градиент достаточно стабилен).
4. **Главный вывод по тестированию архитектуры:**
   качество MLP-сети существенно зависит от **взаимодействия** всех трёх
   гиперпараметров, а не от каждого по отдельности. Перебор в духе
   эксперимента нужен именно потому, что наивная подстановка «больше
   нейронов = выше точность» не выполняется.

## Установка и запуск

### Требования

| Инструмент | Версия |
|------------|--------|
| Python | **3.12** (фиксировано в `.python-version`; диапазон `>=3.12,<3.14`) |
| [`uv`](https://docs.astral.sh/uv/) | актуальная |

Python 3.12 выбран для гарантированной совместимости с PyTorch
(на момент работы у `torch>=2.4` ещё нет официальных wheels для 3.14).

### 1. Установка `uv`

macOS / Linux:

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
# или
brew install uv
```

Windows (PowerShell):

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

### 2. Установка зависимостей

```bash
cd testing_and_debugging_methods/2025/4
uv sync
```

`uv` сам поставит Python 3.12 (если ещё не установлен), создаст `.venv/`
и подтянет `torch`, `numpy`, `pillow`, `pytest`.

### 3. Запуск всех 18 экспериментов

```bash
uv run python -m fourth_laboratory
```

Параметры CLI (всё опционально):

```bash
uv run python -m fourth_laboratory \
    --epochs 10 \
    --n-train 800 \
    --n-test 200 \
    --image-size 28 \
    --device auto \
    --output results.json
```

Доступные значения `--device`: `auto`, `cpu`, `cuda`, `mps`.
По умолчанию выбирается лучшее доступное устройство.

### 4. Запуск pytest

```bash
uv run pytest
```

Ожидаемый результат: `40 passed`.

Запуск отдельных групп:

```bash
uv run pytest tests/unit/test_dataset.py -v
uv run pytest tests/unit/test_model.py -v
uv run pytest tests/unit/test_train.py -v
```

### 5. Быстрый smoke-запуск из Python

```python
from fourth_laboratory.dataset import build_dataset
from fourth_laboratory.model import ShapeClassifier
from fourth_laboratory.train import train_model, TrainConfig

train_ds, test_ds = build_dataset(n_per_class_train=200, n_per_class_test=50, seed=0)
model = ShapeClassifier(input_dim=28 * 28, hidden_dim=100, activation="relu")
result = train_model(model, train_ds, test_ds, TrainConfig(epochs=5, batch_size=100))

print(result.test_metrics)
# EvalMetrics(accuracy=..., mse=..., mae=...)
```

## Выводы

1. Реализована полносвязная MLP-архитектура на **PyTorch** —
   эквивалент `Dense → Activation → Dense` из Keras.
2. Подготовлен **воспроизводимый** программный генератор датасета,
   что снимает зависимость от внешнего Google Drive.
3. Проведены все **18 экспериментов** по заданию, результаты сведены
   в таблицу. Лучшее значение accuracy — **0.8583**, при
   `hidden=5000 / relu / batch_size=100`.
4. Подтверждены типовые наблюдения теории МЛ:
    * ReLU существенно лучше линейной активации в скрытом слое;
    * больший слой помогает только при «здоровом» режиме обучения
      (умеренный `batch_size`);
    * слишком большие батчи режут число шагов оптимизатора и
      ухудшают результат.
5. Покрытие unit-тестами включает корректность форм, типов, шейпов
   и базовую обучаемость модели — это и есть «тестирование
   архитектуры», заявленное в цели лабораторной.
